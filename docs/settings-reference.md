# Settings Reference

This document explains every tunable setting in the model router — what it controls, the value to use, and why.

---

## Shell function env vars (`~/.zshrc`)

These are set inside the `claude-router()` shell function and apply only to sessions started with `claude-router`, not plain `claude`.

### `CLAUDE_CODE_MAX_OUTPUT_TOKENS`

**Current value:** `200000`

Claude Code applies `min(this_value, model_known_max)` and sends the result as `max_tokens` in each request. For `claude-sonnet-4-6`, the model-known max is 128K, so Claude Code always requests `min(200000, 128000) = 128000`. The 200000 value is essentially a no-op on the request side.

**Why not set it lower (e.g. 64000)?**
Qwen BPE tokens ≠ Anthropic tokens. The same content produces roughly 10–30% more Anthropic tokens than Qwen tokens for equivalent text. The router's Qwen ceiling is 65536 Qwen output tokens, which Claude Code's accept-check might see as slightly more than 65536 Anthropic tokens due to drift. Setting `CLAUDE_CODE_MAX_OUTPUT_TOKENS` high prevents Claude Code from rejecting a valid response at the client side.

**Do not set below `128000`.** Claude Code would then request fewer than 128K, and the router's dynamic cap would further reduce it based on remaining context — resulting in unnecessary output truncation.

---

### `CLAUDE_CODE_AUTO_COMPACT_WINDOW`

**Current value:** `262144`

Tells Claude Code the size of the context window. Claude Code uses this to decide when to compact the conversation history. It does **not** affect what the router or model see — it only controls the client-side compaction trigger.

**Why 262144 and not 200000 (Sonnet's native window)?**

| Value | Compact trigger | Post-compact headroom |
|---|---|---|
| 200000 | 55% × 200K = 110K | 90K remaining |
| 262144 | 55% × 262K = 144K | 118K remaining |

Qwen on Bedrock genuinely has a 262K context window. Setting this to 200K would waste 62K of real capacity by triggering compaction too early. Setting it higher than 262K would delay compaction past what Bedrock can handle, causing overflow errors.

**Rule of thumb:** always set this to the actual `context_window` of the largest Bedrock route in `router_config.json`. If you add a model with a different context size, update this value to match.

---

### `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE`

**Current value:** `55`

Compaction fires when conversation history reaches `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE`% of `CLAUDE_CODE_AUTO_COMPACT_WINDOW`. At 55%:

```
55% × 262144 = 144179 tokens → compact trigger
262144 - 144179 = 117965 tokens of remaining headroom after compact
```

**Why 55%?** It's a balance between preserving conversation context (higher %) and leaving enough room for a full multi-turn coding session after compaction (lower %). Values below 40% compact too aggressively; values above 70% leave insufficient headroom and risk overflow in long turns.

Plain `claude` with claude-sonnet-4-6 compacts at the same 55% of its 200K window (110K trigger), so this percentage is consistent with Anthropic's default behavior.

---

### `ANTHROPIC_MODEL`

**Current value:** `opusplan`

Sets the default model for the session. `opusplan` is a Claude Code built-in preset: Opus 4.8 for plan-mode (complex reasoning), Sonnet 4.6 for normal execution turns, Haiku 4.5 for background tasks. These map to the `opus`, `sonnet`, and `haiku` routes in `router_config.json` respectively.

Override per session with `claude-router --model sonnet` or `/model` inside a running session.

---

### `DISABLE_NON_ESSENTIAL_MODEL_CALLS`

**Current value:** `1`

Suppresses Claude Code's background model calls (e.g. automatic PR summaries, inline suggestions). Useful for Bedrock routes where every call has latency and cost. Disable by removing this line if you want Claude Code's full feature set.

## Router config (`~/model-router/router_config.json`)

Generated by the installer. Do not edit by hand — re-run the installer to update.

### `context_window`

Per-route context window in tokens. Used by the router's dynamic cap to estimate how much of the window is consumed by the input and cap `max_tokens` accordingly.

**Must match the actual model limit.** If this is set too high, the router's dynamic cap won't fire soon enough and Bedrock will reject requests with overflow errors.

### `max_tokens`

Per-route hard ceiling on output tokens (in the upstream model's token space — Qwen BPE for Bedrock routes, Anthropic tokens for passthrough routes).

| Route | Value | Notes |
|---|---|---|
| haiku (Qwen 30B) | `32768` | Qwen 30B model limit |
| sonnet (Qwen Coder Next) | `65536` | Qwen Coder Next model limit |
| opus (Anthropic direct) | — | No ceiling; Anthropic enforces server-side |
| fable (Anthropic direct) | — | No ceiling; Anthropic enforces server-side |

The router applies `min(route_max_tokens, remaining_context)` on every request, so the actual output per turn is always bounded by both this ceiling and whatever context space remains after the input.

---

## Dynamic context cap (in `router.py`)

Not a user-configurable setting, but important to understand when debugging.

The router estimates input token count as:

```
estimated_tokens = len(request_body_json_str) / 3.5 * 1.20
```

- `/3.5` converts characters to tokens (rough average for code + English prose).
- `× 1.20` is a 20% safety margin for tokenizer drift and dense content (nested JSON, base64, etc.).

`remaining_tokens = context_window - estimated_tokens`

If `remaining_tokens > 0`, the router caps `max_tokens` to `min(route_max_tokens, remaining_tokens)`.

If `remaining_tokens <= 0` (estimate overshot — common with many MCP tool schemas), the router falls back to a raw estimate (`/ 1.20`) and seeds `max_tokens = raw_remaining × 70%`. This keeps the retry loop from starting at 128K and halving 4 times to find a working value.

**Context overflow retries:** if Bedrock still rejects with an overflow error, the router retries up to 4 times, halving `max_tokens` each attempt. The pre-seeded value means retries usually converge in 1–2 attempts rather than 4.

---

## Why output token counts in logs can look low

When you see `out=61` or `out=122` in `router.log`, it doesn't indicate truncation — those are complete responses that happened to be short. The model generates fewer tokens when:

- The conversation context is nearly full, leaving little room for output (the dynamic cap is working correctly).
- The turn itself only needed a short reply (an acknowledgment, a quick answer, a tool call).

If you see the model cut off mid-sentence, that indicates a `max_tokens` ceiling was hit. Check the log line for the request: if `dynamic max_tokens: capped X → Y` appears, Y is the effective ceiling for that turn.

After a `/compact`, the context resets to a summary (~60–80K tokens), restoring ~180K of output headroom, and responses return to full length.
